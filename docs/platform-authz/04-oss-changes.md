# OSS Changes

This document describes the changes needed in the ToolHive OSS repo to support
platform authorization. These changes are additive and backward-compatible —
existing Cedar policies continue to work without modification.

Refer to [00-invariants.md](00-invariants.md) for design constraints and
[02-cedar-compilation.md](02-cedar-compilation.md) for the policy shapes these
changes must support.

## 1. Add `GroupClaim` + `RoleClaim` to `ConfigOptions`

**File**: `pkg/authz/authorizers/cedar/core.go`

**Current** (line 128):
```go
type ConfigOptions struct {
    Policies     []string `json:"policies" yaml:"policies"`
    EntitiesJSON string   `json:"entities_json" yaml:"entities_json"`
}
```

**Proposed**:
```go
type ConfigOptions struct {
    Policies     []string `json:"policies" yaml:"policies"`
    EntitiesJSON string   `json:"entities_json" yaml:"entities_json"`
    GroupClaim   string   `json:"group_claim,omitempty" yaml:"group_claim,omitempty"`
    RoleClaim    string   `json:"role_claim,omitempty" yaml:"role_claim,omitempty"`
}
```

**Backward compatibility**: `omitempty` + JSON unmarshalling means existing
configs without these fields produce empty strings. The authorizer treats
empty `GroupClaim` as "no group extraction" and empty `RoleClaim` as "no role
extraction" — current behavior preserved.

The `groups` field in `PrincipalCondition` maps to `GroupClaim`, and the
`roles` field maps to `RoleClaim`. Both extract JWT claim values and produce
Cedar `Group` entities. See [05-claim-mapping.md](05-claim-mapping.md) for
the claim resolution logic.

**ConfigMap format** (generated by enterprise controller):
```json
{
  "version": "1.0",
  "type": "cedarv1",
  "cedar": {
    "policies": ["..."],
    "entities_json": "...",
    "group_claim": "groups",
    "role_claim": "roles"
  }
}
```

## 4. Extract groups/roles and set Client parents

**File**: `pkg/authz/authorizers/cedar/core.go`

When `groupClaim` or `roleClaim` is configured, the authorizer must extract
values from the JWT claims and create `Group` parent entities for the `Client`.
Both claim sources produce the same Cedar entity type (`Group`); the
distinction is which JWT claim is read.

**Where**: In `AuthorizeWithJWTClaims` (line 572), after extracting claims and
before calling the `authorize*` methods.

**Proposed helper**:
```go
// extractGroups extracts group names from JWT claims using the configured
// group claim name. Supports dot-notation for nested claims (e.g.,
// "realm_access.roles" for Keycloak).
//
// Returns nil if groupClaim is empty or the claim is not present.
func extractGroups(claims jwt.MapClaims, groupClaim string) []string {
    if groupClaim == "" {
        return nil
    }

    // Support dot-notation for nested claims
    value := resolveNestedClaim(claims, groupClaim)
    if value == nil {
        return nil
    }

    // Convert to string slice
    switch v := value.(type) {
    case []interface{}:
        groups := make([]string, 0, len(v))
        for _, item := range v {
            if s, ok := item.(string); ok {
                groups = append(groups, s)
            }
        }
        return groups
    case []string:
        return v
    default:
        return nil
    }
}

// resolveNestedClaim resolves a dot-separated claim path.
// For example, "realm_access.roles" resolves claims["realm_access"]["roles"].
//
// Auth0 uses namespace-prefixed URLs as claim names (e.g.,
// "https://myapp.example.com/roles") which contain dots. To handle this,
// the resolver tries an exact top-level match first before falling back to
// dot-notation traversal.
func resolveNestedClaim(claims jwt.MapClaims, path string) interface{} {
    // Fast path: exact top-level match (handles Auth0 URLs with dots)
    if v, exists := claims[path]; exists {
        return v
    }

    // Slow path: dot-notation traversal (handles Keycloak nesting)
    parts := strings.Split(path, ".")
    if len(parts) == 1 {
        return nil // single segment already tried above
    }

    var current interface{} = map[string]interface{}(claims)
    for _, part := range parts {
        m, ok := current.(map[string]interface{})
        if !ok {
            return nil
        }
        current = m[part]
    }
    return current
}
```

**Ordering**: `extractGroups` must be called on the **raw** `claims` before
`preprocessClaims` adds the `claim_` prefix. The `groupClaim` value (e.g.,
`"groups"`) matches the raw JWT claim name, not the preprocessed name
`"claim_groups"`.

**How groups flow into entities**: Groups are extracted once in
`AuthorizeWithJWTClaims` and threaded to the `authorize*` methods, which
pass them to `CreateEntitiesForRequest`. The `serverName` is already on the
`Authorizer` struct (change 2), so the `authorize*` methods access it via
`a.serverName`.

```go
func (a *Authorizer) AuthorizeWithJWTClaims(
    ctx context.Context,
    feature authorizers.MCPFeature,
    operation authorizers.MCPOperation,
    resourceID string,
    arguments map[string]interface{},
) (bool, error) {
    // ... extract identity, clientID, claims (existing code) ...

    // NEW: extract groups and roles from raw claims BEFORE preprocessing
    groups := extractGroups(claims, a.groupClaim)
    roles := extractGroups(claims, a.roleClaim)
    allGroups := dedup(append(groups, roles...))

    // Preprocess claims and arguments (existing code)
    processedClaims := preprocessClaims(claims)
    processedArgs := preprocessArguments(arguments)

    // Pass unified groups to authorize methods
    switch {
    case feature == authorizers.MCPFeatureTool && operation == authorizers.MCPOperationCall:
        return a.authorizeToolCall(clientID, resourceID, processedClaims, processedArgs, allGroups)
    // ... other cases with groups parameter ...
    }
}
```

Each `authorize*` method gains a `groups []string` parameter and passes it
(along with `a.serverName`) to `CreateEntitiesForRequest`. The 4 methods
that need updating are: `authorizeToolCall`, `authorizePromptGet`,
`authorizeResourceRead`, and `authorizeFeatureList`.

**`CreateEntitiesForRequest` changes**: When groups are provided, the method
creates `Group` parent UIDs and passes them to `CreatePrincipalEntity`. It
also creates intermediate `Group` entities (with empty parents — the static
entities from `entities_json` carry the `Role` parents):

```go
func (f *EntityFactory) CreateEntitiesForRequest(
    principal, action, resource string,
    claimsMap map[string]interface{},
    attributes map[string]interface{},
    groups []string,                      // NEW parameter
    serverName string,                    // NEW parameter
) (cedar.EntityMap, error) {
    // ... parse principal, action, resource (existing) ...

    entities := make(cedar.EntityMap)

    // Build group parent UIDs for the principal
    var groupParents []cedar.EntityUID
    for _, g := range groups {
        groupUID := cedar.NewEntityUID("Group", cedar.String(g))
        groupParents = append(groupParents, groupUID)

        // Create Group entity (parents come from static entities_json via merge)
        entities[groupUID] = cedar.Entity{
            UID:        groupUID,
            Parents:    cedar.NewEntityUIDSet(),
            Attributes: cedar.NewRecord(cedar.RecordMap{}),
            Tags:       cedar.NewRecord(cedar.RecordMap{}),
        }
    }

    // Create principal entity with group parents
    principalUID, principalEntity := f.CreatePrincipalEntity(
        principalType, principalID, claimsMap, groupParents...)
    entities[principalUID] = principalEntity

    // Create action entity (unchanged)
    actionUID, actionEntity := f.CreateActionEntity(actionType, actionID, nil)
    entities[actionUID] = actionEntity

    // Create resource entity with MCP parent (see change 5)
    var resourceParents []cedar.EntityUID
    if serverName != "" {
        mcpUID := cedar.NewEntityUID("MCP", cedar.String(serverName))
        resourceParents = append(resourceParents, mcpUID)
    }
    resourceUID, resourceEntity := f.CreateResourceEntity(
        resourceType, resourceID, attributes, resourceParents...)
    entities[resourceUID] = resourceEntity

    return entities, nil
}
```

**Why Group entities have empty parents here**: The dynamic `Group` entities
created per-request have no parents set. The static `Group` entities in
`entities_json` carry the `Role` parents (e.g., `Group::"platform-eng"` →
`Role::"writer"`). During entity merging, the static entities are loaded
first, then dynamic entities override. Since the dynamic `Group` entity has
the same UID as the static one, the static version (with Role parents) is
the one that survives in the merged map — **unless** the dynamic one
overwrites it.

**Important**: The merge order matters. Currently `core.go` (line 293-308)
copies static entities first, then overlays dynamic entities. This means
dynamic entities **override** static ones. For `Group` entities, this is
wrong — the dynamic `Group` (empty parents) would overwrite the static
`Group` (with Role parents).

**Fix**: The `CreateEntitiesForRequest` method should **not** create dynamic
`Group` entities. Instead, it should only create the `Client` entity with
`Group` parent UIDs. The `Group` entities already exist in the static entity
store with the correct `Role` parents. Cedar's `in` operator traverses the
full entity DAG — it will find `Client → Group → Role` as long as:
1. The `Client` entity lists `Group` UIDs as parents
2. The `Group` entity exists (from static store) with `Role` parents

**Corrected approach**:
```go
// Build group parent UIDs for the principal (no Group entities created)
var groupParents []cedar.EntityUID
for _, g := range groups {
    groupParents = append(groupParents,
        cedar.NewEntityUID("Group", cedar.String(g)))
}

// Create principal entity with group parents
principalUID, principalEntity := f.CreatePrincipalEntity(
    principalType, principalID, claimsMap, groupParents...)
entities[principalUID] = principalEntity
```

If a group from the JWT has no matching static `Group` entity, Cedar does
not error. It treats the missing entity as a leaf node with no parents —
hierarchy traversal stops and the group contributes no role membership.
Other valid group paths still work. This is the graceful degradation
described in
[02-cedar-compilation.md](02-cedar-compilation.md#missing-groups-degrade-gracefully).

## 6. Support `readOnlyHint` on Tool entities

**File**: `pkg/authz/authorizers/cedar/core.go`

The reader role's `call_tool` policy uses:
```cedar
resource has readOnlyHint && resource.readOnlyHint == true
```

The `readOnlyHint` attribute must be present on `Tool` entities for this to
work. The attribute comes from MCP tool annotations (MCP spec 2025-03-26+).

**Current** `authorizeToolCall` (line 392):
```go
attributes := mergeContexts(map[string]interface{}{
    "name":      toolName,
    "operation": "call",
    "feature":   "tool",
}, attrsMap)
```

**Prefix problem**: The `attrsMap` parameter in `authorizeToolCall` receives
`processedArgs` from `AuthorizeWithJWTClaims`. The `preprocessArguments`
function (line 355) adds an `arg_` prefix to all keys, so `readOnlyHint`
would become `arg_readOnlyHint` on the entity. The Cedar policy expects
`resource has readOnlyHint` (unprefixed), so this path does not work.

**Fix**: `readOnlyHint` is a **tool annotation**, not a user argument. It
must bypass `preprocessArguments` and be set directly on the resource entity
attributes. The `authorizeToolCall` method should accept a separate
`annotations` map:

```go
func (a *Authorizer) authorizeToolCall(
    clientID, toolName string,
    claimsMap map[string]interface{},
    attrsMap map[string]interface{},
    groups []string,
    annotations map[string]interface{},  // NEW: tool annotations (readOnlyHint, etc.)
) (bool, error) {
    // ...
    // Merge base attributes with annotations (unprefixed) and processed args
    attributes := mergeContexts(map[string]interface{}{
        "name":      toolName,
        "operation": "call",
        "feature":   "tool",
    }, annotations, attrsMap)
    // ...
}
```

The caller in `AuthorizeWithJWTClaims` passes annotations separately:

```go
case feature == authorizers.MCPFeatureTool && operation == authorizers.MCPOperationCall:
    annotations := extractToolAnnotations(arguments)  // pull readOnlyHint etc.
    return a.authorizeToolCall(clientID, resourceID, processedClaims,
        processedArgs, groups, annotations)
```

**No change to entity.go needed** — `convertToCedarValue` already handles
`bool` values (line 150). The attribute flows through
`CreateResourceEntity` → `convertMapToCedarRecord` → `convertToCedarValue`
→ `cedar.True`/`cedar.False`.

**When `readOnlyHint` is unavailable**: The `annotations` map is nil or
empty. The attribute is not set on the entity, so `resource has readOnlyHint`
returns false and the reader's `call_tool` policy does not match. This is
the intended cold-start behavior documented in
[02-cedar-compilation.md](02-cedar-compilation.md#resource-entity-tool-prompt-resource).

**Where the caller gets annotations**: The MCP transport layer caches tool
metadata from `tools/list` responses. The `readOnlyHint` value is extracted
from the tool's `annotations` field in the MCP response. The caching
mechanism is outside the scope of the Cedar authorizer — it is the caller's
responsibility to pass the annotation when available.

## 7. Replace `FeatureType` with `MCP` for list operations

**File**: `pkg/authz/authorizers/cedar/core.go`

This change is **conditional** — behavior only changes when `serverName` is
configured. Existing standalone deployments are unaffected.

**Current** `authorizeFeatureList` (line 499):
```go
func (a *Authorizer) authorizeFeatureList(
    clientID string,
    feature authorizers.MCPFeature,
    claimsMap map[string]interface{},
    attrsMap map[string]interface{},
) (bool, error) {
    principal := fmt.Sprintf("Client::%s", clientID)
    action := fmt.Sprintf("Action::list_%ss", feature)
    resource := fmt.Sprintf("FeatureType::%s", feature)  // <-- problem
    // ...
}
```

The `FeatureType` entity type is not part of the Cedar schema defined in
[02-cedar-compilation.md](02-cedar-compilation.md#7-cedar-schema). Enterprise
policies use `resource in MCP::"github"` for server-scoped list operations.
A `FeatureType::tool` resource will never match `resource in MCP::"github"`.

**Proposed**: Use the `MCP` server entity as the resource for list operations
when `serverName` is configured:

```go
func (a *Authorizer) authorizeFeatureList(
    clientID string,
    feature authorizers.MCPFeature,
    claimsMap map[string]interface{},
    attrsMap map[string]interface{},
    groups []string,
) (bool, error) {
    principal := fmt.Sprintf("Client::%s", clientID)
    action := fmt.Sprintf("Action::list_%ss", feature)

    // Use MCP server entity when serverName is configured (enterprise),
    // fall back to FeatureType for backward compatibility (standalone)
    resource := func() string {
        if a.serverName != "" {
            return fmt.Sprintf("MCP::%s", a.serverName)
        }
        return fmt.Sprintf("FeatureType::%s", feature)
    }()

    // ...
}
```

**Backward compatibility**: When `serverName` is empty (standalone Cedar
usage), the resource remains `FeatureType::<feature>` — existing policies
that use `FeatureType` continue to work. When `serverName` is set (enterprise
usage), the resource is `MCP::<server>` — matching the compiled policy shapes.

**Policy migration**: Existing standalone Cedar policies that reference
`FeatureType::tool` are unaffected. Enterprise policies are always generated
by the controller and use `MCP::<server>`. There is no migration needed.

## 8. Rename `uri` attribute to `name` on Resource entities

**File**: `pkg/authz/authorizers/cedar/core.go`

**Current** `authorizeResourceRead` (line ~437):
```go
attributes := mergeContexts(map[string]interface{}{
    "uri":       resourceURI,
    "operation": "read",
    "feature":   "resource",
}, attrsMap)
```

The `Resource` entity currently exposes the URI under the attribute key `"uri"`.
The Cedar schema defined in
[02-cedar-compilation.md](02-cedar-compilation.md#7-cedar-schema) declares the
attribute as `"name"` (consistent with `Tool` and `Prompt` entities). A policy
that references `resource.uri` will silently fail schema validation under the
enterprise schema.

**Proposed**: Add `"name"` as the primary attribute and keep `"uri"` as an
alias for backward compatibility:

```go
attributes := mergeContexts(map[string]interface{}{
    "name":      resourceURI,    // canonical name used by Cedar schema
    "uri":       resourceURI,    // backward-compatible alias
    "operation": "read",
    "feature":   "resource",
}, attrsMap)
```

**Backward compatibility**: Existing standalone Cedar policies that reference
`resource.uri` continue to work because the alias is preserved. Enterprise
policies generated by the controller use `resource.name`. Both attributes
carry the same value.

**Why `name` and not `uri` in the schema**: Cedar schema uses `name` for all
three resource types (`Tool`, `Prompt`, `Resource`) for consistency. The
resource URI is the identifier of the resource, analogous to the tool name for
`Tool` entities. Renaming to `name` in the schema avoids a type-specific
special case.

## Dependency Order

The changes should be implemented in this order due to dependencies:

```
1. ConfigOptions.GroupClaim      (standalone, no deps)
2. Authorizer.serverName         (standalone, no deps)
3. Entity factory parents        (standalone, no deps)
4. Group extraction + Client parents  (depends on 1, 2, 3)
5. MCP parent on resources       (depends on 2, 3)
6. readOnlyHint support          (standalone, no deps)
7. FeatureType → MCP for lists   (depends on 2)
8. Resource entity uri→name      (standalone, no deps)
```

Changes 1, 2, 3, 6, and 8 can be done in parallel (independent). Changes 4,
5, and 7 depend on earlier changes.

## Compatibility and Migration

After all 8 changes, existing standalone Cedar policies are largely drop-in
compatible with the enterprise entity model. This table summarises which
constructs require rewriting when moving policies to `rawPolicies` in a
`ToolhiveAuthorizationPolicy`:

| Construct | Compatible? | Notes |
|-----------|-------------|-------|
| `principal == Client::"<sub>"` | Yes | Unchanged |
| `principal in Role::"<name>"` | Yes — after C-4 | Client→Group→Role parents populated by C-4 |
| `action == Action::"call_tool"` | Yes | Unchanged |
| `resource == Tool::"<name>"` | Yes | Unchanged |
| `resource == Prompt::"<name>"` | Yes | Unchanged |
| `resource == Resource::"<name>"` | Yes | URI used as entity ID; unchanged |
| `resource.uri` | Yes — alias | Both `uri` and `name` are set (C-8) |
| `resource in MCP::"<server>"` | Yes — after C-5 | Resource entities get MCP parent |
| `context.ip_address` etc. | Yes | Unchanged |
| `resource == FeatureType::"tool"` | **No** | The only rewrite required — see below |

### The `FeatureType` incompatibility

The only policy construct that is **not** drop-in compatible is the use of
`FeatureType::"tool"` as the resource in list-operation policies. This entity
type is synthetic OSS scaffolding that has no equivalent in the enterprise
Cedar schema.

**Before (standalone):**
```cedar
permit(
  principal == Client::"alice",
  action == Action::"list_tools",
  resource == FeatureType::"tool"
);
```

**After (enterprise, in `rawPolicies`):**
```cedar
permit(
  principal == Client::"alice",
  action == Action::"list_tools",
  resource in MCP::"github"
);
```

The resource changes from the synthetic `FeatureType` singleton to the actual
`MCP` server container. All other parts of the policy are identical.

### Why `FeatureType` cannot be made compatible

Making `FeatureType::"tool"` a child of `MCP::"<server>"` so that
`resource in MCP::"github"` matches it was evaluated and rejected. The problem
is that `FeatureType::"tool"` is a **singleton entity** — there is one instance
regardless of how many servers are running. In a vMCP deployment where a single
shared Cedar authorizer evaluates requests across multiple backends, the
singleton cannot simultaneously be a child of `MCP::"github"` and
`MCP::"atlassian"`. Server isolation — the key security property of the
enterprise model — would be silently broken.
