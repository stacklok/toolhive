# Define `MCPTelemetryConfig` API Types with `telemetry.Config` Embedding

**Parent**: [STORY-03](STORY-03.md) — MCPTelemetryConfig CRD

## Description

Create the `MCPTelemetryConfig` CRD type definitions that embed the canonical `telemetry.Config` type from `pkg/telemetry/config.go`. This eliminates the type divergence between CRD and application config that has caused conversion bugs.

## Implementation Notes

**New file**: `cmd/thv-operator/api/v1alpha2/mcptelemetryconfig_types.go`

**Type embedding approach**: Use a wrapper struct that embeds `telemetry.Config` but excludes CLI-only fields:

```go
type MCPTelemetryConfigSpec struct {
    // Embed the canonical telemetry config type.
    // CLI-only fields (EnvironmentVariables, CustomAttributes) are excluded via json:"-".
    telemetry.Config `json:",inline"`

    // SensitiveHeaders holds credential headers via SecretKeyRef.
    // Never store tokens as plaintext in CRD spec.
    SensitiveHeaders []SensitiveHeader `json:"sensitiveHeaders,omitempty"`
}

type SensitiveHeader struct {
    // HeaderName is the HTTP header name (e.g., "Authorization").
    HeaderName string `json:"headerName"`
    // ValueSecretRef references a Secret key containing the header value.
    ValueSecretRef SecretKeySelector `json:"valueSecretRef"`
}
```

**Key type decisions**:
- `Headers` is `map[string]string` — this is the `telemetry.Config` native format. The existing CRD `OpenTelemetryConfig` uses `[]string` (key=value pairs), which is the breaking change.
- `EnvironmentVariables` and `CustomAttributes` must be excluded — they are CLI-mode only. Use `json:"-"` tags on the embedded fields, or use a dedicated wrapper struct that only includes CRD-relevant fields.
- `ServiceName` is excluded from this CRD — it is a per-server field at the reference site.

**Precedent**: `VirtualMCPServer` already embeds `config.Config` from `pkg/vmcp/config/config.go` directly in its CRD spec (`virtualmcpserver_types.go:68`). This proves the embedding pattern works with kubebuilder and deepcopy generation.

**Status** (same pattern as MCPOIDCConfig):
```go
type MCPTelemetryConfigStatus struct {
    ObservedGeneration int64              `json:"observedGeneration,omitempty"`
    ConfigHash         string             `json:"configHash,omitempty"`
    ReferencingServers []string           `json:"referencingServers,omitempty"`
    Conditions         []metav1.Condition `json:"conditions,omitempty"`
}
```

**Potential risk**: Embedding `telemetry.Config` may surface kubebuilder marker or deepcopy issues. If `json:"-"` tags on embedded struct fields don't work cleanly with controller-gen, fall back to a purpose-built struct that mirrors `telemetry.Config` fields (minus CLI-only ones) with a simple conversion function.

**Post-implementation**: Run `task gen` to generate deepcopy methods and CRD manifests.

## Testing

**Type embedding tests**:
- Verify CLI-only fields (`EnvironmentVariables`, `CustomAttributes`) are not serialized to JSON
- Verify `Headers` is `map[string]string` (not `[]string`)
- Verify `SensitiveHeaders` serializes correctly with `SecretKeyRef`
