# Implement `MCPTelemetryConfig` Controller

**Parent**: [STORY-03](STORY-03.md) — MCPTelemetryConfig CRD

## Description

Implement the controller for `MCPTelemetryConfig` following the same pattern as the MCPOIDCConfig controller (02-C). The controller handles reference tracking, finalizer-based deletion protection, config change cascade, and Secret resolution for sensitive headers.

## Implementation Notes

**New file**: `cmd/thv-operator/controllers/mcptelemetryconfig_controller.go`

**Controller responsibilities** (same as MCPOIDCConfig controller):

1. **Finalizer management**: `mcptelemetryconfig.toolhive.stacklok.dev/finalizer`
2. **Reference tracking**: Scan MCPServer and (when available) MCPRemoteEndpoint for `spec.telemetryConfig.ref.name == this.name`. Use `{Kind}/{Name}` format for entries (e.g., `MCPServer/my-server`). Also update `status.referenceCount` to `len(referencingServers)`.
3. **Config hash**: Compute SHA-256 hash of resolved config. For `sensitiveHeaders`, resolve Secret values to include in hash computation (so hash changes when Secret content changes), but do not expose Secret values in status.
4. **Cascade reconciliation**: Write annotation `toolhive.stacklok.dev/telemetryconfig-hash: <newHash>` on referencing workloads
5. **Deletion protection**: Block finalizer removal while `referencingServers` is non-empty
6. **Status conditions**: Set `Ready` condition. Validate that referenced Secrets for `sensitiveHeaders` exist.
7. **Watches**: MCPServer and (when available) MCPRemoteEndpoint via `EnqueueRequestsFromMapFunc`. Additionally, watch Secrets referenced by `sensitiveHeaders` so that changes to Secret content trigger re-reconciliation and config hash recomputation.

> **Note on MCPRemoteEndpoint**: MCPRemoteEndpoint CRD types come from RFC-0057 and may not be merged yet. If the types are not available, omit MCPRemoteEndpoint watches and reference tracking initially. STORY-07 sub-issue 07-C will add them when MCPRemoteEndpoint lands.

**Required RBAC markers** (add to the reconciler):
```go
//+kubebuilder:rbac:groups=toolhive.stacklok.dev,resources=mcptelemetryconfigs,verbs=get;list;watch;create;update;patch;delete
//+kubebuilder:rbac:groups=toolhive.stacklok.dev,resources=mcptelemetryconfigs/status,verbs=get;update;patch
//+kubebuilder:rbac:groups=toolhive.stacklok.dev,resources=mcptelemetryconfigs/finalizers,verbs=update
//+kubebuilder:rbac:groups=toolhive.stacklok.dev,resources=mcpservers,verbs=get;list;watch;patch
//+kubebuilder:rbac:groups=toolhive.stacklok.dev,resources=mcpremoteendpoints,verbs=get;list;watch;patch
//+kubebuilder:rbac:groups="",resources=secrets,verbs=get;list;watch
```

**Secret resolution for sensitive headers**:
- For each `sensitiveHeader`, resolve the `valueSecretRef` to verify the Secret and key exist
- Include resolved values in hash computation for change detection
- If a referenced Secret does not exist, set `Ready=False` with reason `SecretNotFound`

**Note**: VirtualMCPServer does not reference MCPTelemetryConfig (telemetry for vMCP is configured differently via embedded `config.Config`).

## Testing

**Unit tests** for controller reconciliation:
- Happy path: create MCPTelemetryConfig → verify finalizer, `Ready` condition, `configHash`
- Config change cascade: update endpoint → verify `configHash` changes, annotations written
- Sensitive header resolution: Secret exists → `Ready=True`. Secret missing → `Ready=False`, reason `SecretNotFound`
- Finalizer lifecycle: add on first reconcile, remove when no references
- Deletion protection: references exist → blocked
- Reference tracking: create/delete MCPServer → verify `referencingServers` updated

**Integration tests** in `cmd/thv-operator/test-integration/`:

Reference lifecycle:
1. Create MCPTelemetryConfig with valid config → verify Ready
2. Create MCPServer referencing it with `serviceName: "my-server"` → verify `referencingServers` updated
3. Update MCPTelemetryConfig endpoint → verify cascade annotation on MCPServer
4. Delete MCPServer → verify `referencingServers` updated
5. Delete MCPTelemetryConfig → verify deletion succeeds

Sensitive header lifecycle:
1. Create Secret with auth token
2. Create MCPTelemetryConfig with `sensitiveHeaders` referencing the Secret → verify Ready
3. Delete the Secret → reconcile → verify `Ready=False`
4. Recreate the Secret → reconcile → verify `Ready=True`

Deletion protection:
1. Create MCPTelemetryConfig + MCPServer referencing it
2. Delete MCPTelemetryConfig → verify stuck in Terminating
3. Delete MCPServer → verify MCPTelemetryConfig deletion proceeds

## Acceptance Criteria

- [ ] Controller registered and reconciles MCPTelemetryConfig resources
- [ ] Finalizer lifecycle matches MCPOIDCConfig pattern
- [ ] `status.referencingServers` tracks MCPServer and MCPRemoteEndpoint references
- [ ] `status.configHash` includes resolved sensitive header values
- [ ] Config changes cascade via annotation to referencing workloads
- [ ] Deletion blocked while references exist
- [ ] `Ready=False` with `SecretNotFound` if sensitiveHeader Secret is missing
- [ ] Watches registered for MCPServer, MCPRemoteEndpoint (when available), and Secrets (referenced by `sensitiveHeaders`)
- [ ] RBAC markers added for all resource accesses: `mcptelemetryconfigs`, `mcptelemetryconfigs/status`, `mcpservers`, `mcpremoteendpoints`, `secrets`
- [ ] Unit tests cover controller happy path, error paths, and sensitive header resolution
- [ ] Integration tests cover reference lifecycle, sensitive header lifecycle, and deletion protection
- [ ] All tests pass in CI (`task test`)
