# Implement `MCPOIDCConfig` Controller

**Parent**: [STORY-02](STORY-02.md) — MCPOIDCConfig CRD

## Description

Implement the controller for `MCPOIDCConfig` following the established `MCPExternalAuthConfig` controller pattern. The controller handles reference tracking, finalizer-based deletion protection, config change cascade, and status management.

## Implementation Notes

**New file**: `cmd/thv-operator/controllers/mcpoidcconfig_controller.go`

**Pattern to follow**: `mcpexternalauthconfig_controller.go` — this is the authoritative reference for:
- Finalizer lifecycle (lines 29-30, 72-81)
- Reference tracking via `findReferencingMCPServers()` (lines 239-250)
- Cascade reconciliation via annotations (lines 151-187)
- Deletion protection (lines 192-236)
- Watch setup with `EnqueueRequestsFromMapFunc` (lines 254-282)

**Controller responsibilities**:

1. **Finalizer management**: Add `mcpoidcconfig.toolhive.stacklok.dev/finalizer` on first reconciliation. Remove only when `referencingServers` is empty.

2. **Reference tracking**: `findReferencingWorkloads()` scans MCPServer, VirtualMCPServer, and (when available) MCPRemoteEndpoint for resources with `spec.oidcConfig.ref.name == this.name`. Updates `status.referencingServers` on each reconciliation using `{Kind}/{Name}` format (e.g., `MCPServer/my-server`, `VirtualMCPServer/my-vmcp`). Also updates `status.referenceCount` to `len(referencingServers)`.

3. **Config hash**: Compute SHA-256 hash of the resolved configuration. Store in `status.configHash`.

4. **Cascade reconciliation**: When `configHash` changes, write annotation `toolhive.stacklok.dev/oidcconfig-hash: <newHash>` on each referencing workload. The annotation write triggers the workload controller's reconcile loop.

5. **Deletion protection**: If `referencingServers` is non-empty, block finalizer removal and requeue. Log a message identifying which workloads still reference this config.

6. **Status conditions**: Set `Ready` condition based on config validation (e.g., inline config has required fields, ConfigMap exists, ServiceAccount exists).

7. **Watches**: Register watches for MCPServer, VirtualMCPServer, and (when available) MCPRemoteEndpoint changes via `EnqueueRequestsFromMapFunc` to detect new/removed references.

> **Note on MCPRemoteEndpoint**: MCPRemoteEndpoint CRD types come from RFC-0057 and may not be merged yet. If the types are not available, omit MCPRemoteEndpoint watches and reference tracking initially. STORY-07 sub-issue 07-C will add them when MCPRemoteEndpoint lands.

## Testing

**Unit tests** for controller reconciliation:
- Happy path: create MCPOIDCConfig → verify finalizer added, `Ready` condition set, `configHash` computed
- Config change: update MCPOIDCConfig → verify `configHash` changes, annotations written to referencing workloads
- Finalizer add: first reconcile adds finalizer, requeues
- Finalizer remove: no references → finalizer removed → deletion proceeds
- Deletion protection: references exist → finalizer held → deletion blocked
- Reference tracking: create/delete workloads → verify `referencingServers` updated

**Integration tests** in `cmd/thv-operator/test-integration/`:

Reference lifecycle:
1. Create MCPOIDCConfig → verify Ready
2. Create MCPServer referencing it → verify MCPOIDCConfig `referencingServers` updated
3. Update MCPOIDCConfig → verify cascade annotation written on MCPServer
4. Delete MCPServer → verify `referencingServers` updated
5. Delete MCPOIDCConfig → verify deletion succeeds (no references)

Deletion protection:
1. Create MCPOIDCConfig + MCPServer referencing it
2. Delete MCPOIDCConfig → verify stuck in `Terminating` (finalizer held)
3. Delete MCPServer → verify MCPOIDCConfig deletion proceeds
