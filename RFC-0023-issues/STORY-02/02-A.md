# Define `MCPOIDCConfig` API Types

**Parent**: [STORY-02](STORY-02.md) — MCPOIDCConfig CRD

## Description

Create the `MCPOIDCConfig` CRD type definitions with CEL-only union validation (no `type` discriminator field). This CRD holds shared OIDC provider configuration that can be referenced by multiple workload CRDs.

## Implementation Notes

**New file**: `cmd/thv-operator/api/v1alpha2/mcpoidcconfig_types.go`

**Spec — three source variants (CEL union)**:
```go
// +kubebuilder:validation:XValidation:rule="(has(self.kubernetesServiceAccount) ? 1 : 0) + (has(self.configMapRef) ? 1 : 0) + (has(self.inline) ? 1 : 0) == 1",message="exactly one of kubernetesServiceAccount, configMapRef, or inline must be set"
type MCPOIDCConfigSpec struct {
    KubernetesServiceAccount *KubernetesServiceAccountOIDC `json:"kubernetesServiceAccount,omitempty"`
    ConfigMapRef             *ConfigMapOIDCSource           `json:"configMapRef,omitempty"`
    Inline                   *InlineOIDCProviderConfig      `json:"inline,omitempty"`
}
```

**Shared fields** (in `InlineOIDCProviderConfig` or resolved from other sources):
- `issuer` (required) — OIDC issuer URL
- `jwksUri` (optional) — discovered from issuer if empty
- `introspectionUrl` (optional)
- `clientId` (required)
- `clientSecretRef` (`SecretKeySelector`) — reference to Secret, no inline plaintext
- `jwksAllowPrivateIP` (bool)
- `insecureAllowHTTP` (bool)
- `caBundleRef` (`CABundleSource`) — ConfigMap reference for CA certificates

**Status**:
```go
type MCPOIDCConfigStatus struct {
    ObservedGeneration int64              `json:"observedGeneration,omitempty"`
    ConfigHash         string             `json:"configHash,omitempty"`
    ReferencingServers []string           `json:"referencingServers,omitempty"`
    Conditions         []metav1.Condition `json:"conditions,omitempty"`
}
```

**Key patterns to follow**:
- `MCPExternalAuthConfig` types at `mcpexternalauthconfig_types.go` for status shape and CEL patterns
- Existing `InlineOIDCConfig` at `mcpserver_types.go:516-592` for field definitions (but exclude `audience`, `scopes`, `clientSecret`, `thvCABundlePath`)

**Post-implementation**: Run `task gen` to generate deepcopy methods and CRD manifests.

## Testing

Integration tests in `cmd/thv-operator/test-integration/` covering all `MCPOIDCConfigSpec` CEL permutations:

- Valid: only `inline` set → accepted
- Valid: only `configMapRef` set → accepted
- Valid: only `kubernetesServiceAccount` set → accepted
- Invalid: multiple variants set → rejected with correct message
- Invalid: no variant set → rejected

Tests should apply manifests against the envtest API server and assert the expected validation error messages.
