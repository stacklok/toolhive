# Define `MCPTelemetryConfig` API Types

**Parent**: [STORY-03](STORY-03.md) — MCPTelemetryConfig CRD

## Description

Create the `MCPTelemetryConfig` CRD type definitions using a purpose-built wrapper struct that mirrors the CRD-relevant fields from `telemetry.Config` (`pkg/telemetry/config.go`). A simple `ToTelemetryConfig()` conversion method bridges the CRD type to the application type. This eliminates the type divergence between CRD and application config that has caused conversion bugs.

## Implementation Notes

**New file**: `cmd/thv-operator/api/v1alpha2/mcptelemetryconfig_types.go`

**Why not embed `telemetry.Config` directly?** Go does not allow overriding JSON struct tags on fields of an embedded struct. `telemetry.Config` contains CLI-only fields (`EnvironmentVariables`, `CustomAttributes`) and a per-server field (`ServiceName`) that must not appear in the CRD spec. Using `json:",inline"` would expose all of these, and `json:"-"` cannot be applied to another struct's fields from the embedding site.

**Recommended approach — purpose-built wrapper struct**:

```go
// MCPTelemetryConfigSpec contains the CRD-relevant subset of telemetry.Config.
// CLI-only fields (EnvironmentVariables, CustomAttributes) and the per-server
// field (ServiceName) are intentionally excluded.
type MCPTelemetryConfigSpec struct {
    Endpoint                    string            `json:"endpoint,omitempty"`
    ServiceVersion              string            `json:"serviceVersion,omitempty"`
    TracingEnabled              bool              `json:"tracingEnabled"`
    MetricsEnabled              bool              `json:"metricsEnabled"`
    SamplingRate                string            `json:"samplingRate,omitempty"`
    Headers                     map[string]string `json:"headers,omitempty"`
    Insecure                    bool              `json:"insecure,omitempty"`
    EnablePrometheusMetricsPath bool              `json:"enablePrometheusMetricsPath,omitempty"`
    UseLegacyAttributes         bool              `json:"useLegacyAttributes"`
    SensitiveHeaders            []SensitiveHeader `json:"sensitiveHeaders,omitempty"`
}

type SensitiveHeader struct {
    HeaderName     string            `json:"headerName"`
    ValueSecretRef SecretKeySelector `json:"valueSecretRef"`
}
```

**Conversion method** (in the same file or a `_conversion.go` file):
```go
func (s *MCPTelemetryConfigSpec) ToTelemetryConfig() telemetry.Config {
    return telemetry.Config{
        Endpoint:                    s.Endpoint,
        ServiceVersion:              s.ServiceVersion,
        TracingEnabled:              s.TracingEnabled,
        MetricsEnabled:              s.MetricsEnabled,
        SamplingRate:                s.SamplingRate,
        Headers:                     s.Headers,
        Insecure:                    s.Insecure,
        EnablePrometheusMetricsPath: s.EnablePrometheusMetricsPath,
        UseLegacyAttributes:         s.UseLegacyAttributes,
    }
}
```

**Key type decisions**:
- `Headers` is `map[string]string` — this is the `telemetry.Config` native format. The existing CRD `OpenTelemetryConfig` uses `[]string` (key=value pairs), which is the breaking change.
- `EnvironmentVariables`, `CustomAttributes`, and `ServiceName` are intentionally absent from the wrapper struct — excluded at the type level, not via struct tags.
- `ServiceName` is excluded from this CRD — it is a per-server field at the reference site.

**Maintenance note**: When fields are added to `telemetry.Config`, a corresponding update to `MCPTelemetryConfigSpec` and `ToTelemetryConfig()` is required for CRD-relevant fields. Add a unit test that compares the field sets to catch drift.

**Status** (same pattern as MCPOIDCConfig):
```go
type MCPTelemetryConfigStatus struct {
    ObservedGeneration int64              `json:"observedGeneration,omitempty"`
    ConfigHash         string             `json:"configHash,omitempty"`
    // ReferencingServers uses {Kind}/{Name} format (e.g., "MCPServer/my-server")
    // to avoid name collisions across referencing kinds.
    ReferencingServers []string           `json:"referencingServers,omitempty"`
    ReferenceCount     int32              `json:"referenceCount,omitempty"`
    Conditions         []metav1.Condition `json:"conditions,omitempty"`
}
```

**Post-implementation**: Run `task gen` to generate deepcopy methods and CRD manifests.

## Testing

**Type and conversion tests**:
- Verify `MCPTelemetryConfigSpec` does not have `EnvironmentVariables`, `CustomAttributes`, or `ServiceName` fields (reflection test)
- Verify `ToTelemetryConfig()` round-trips all CRD fields correctly
- Verify `Headers` is `map[string]string` (not `[]string`)
- Verify `SensitiveHeaders` serializes correctly with `SecretKeyRef`
- **Drift detection**: Enumerate `telemetry.Config` JSON field names and assert that each is either present in `MCPTelemetryConfigSpec` or in an explicit exclusion list (`serviceName`, `environmentVariables`, `customAttributes`)

## Acceptance Criteria

- [ ] `MCPTelemetryConfigSpec` is a purpose-built wrapper struct mirroring `telemetry.Config` (minus CLI-only and per-server fields)
- [ ] `ToTelemetryConfig()` conversion method implemented and tested
- [ ] `Headers` is `map[string]string`
- [ ] `SensitiveHeaders` uses `SecretKeyRef` for credential headers
- [ ] `EnvironmentVariables` and `CustomAttributes` are not present in the CRD spec

- [ ] `ReferencingServers` entries use `{Kind}/{Name}` format to disambiguate across workload types
