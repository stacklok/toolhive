# Update MCPRemoteEndpoint Types and Controller

**Parent**: [STORY-07](STORY-07.md) — Workload CRD Config Reference Updates

## Description

Add OIDC and telemetry config reference fields to MCPRemoteEndpoint (RFC-0057) and update its controller with the same resolution and fail-closed logic as MCPServer.

## Implementation Notes

**Prerequisite**: MCPRemoteEndpoint CRD types must exist (from RFC-0057). If not yet available, this sub-issue should be deferred.

**Type changes**: Add to `MCPRemoteEndpointSpec`:
```go
// OIDCConfig specifies OIDC authentication configuration via reference or inline.
// +optional
OIDCConfig *OIDCConfigWithRef `json:"oidcConfig,omitempty"`

// TelemetryConfig specifies telemetry configuration via reference or inline.
// +optional
TelemetryConfig *TelemetryConfigWithRef `json:"telemetryConfig,omitempty"`
```

**`type: proxy` only**: The `oidcConfig` and `telemetryConfig` fields are only applicable when `spec.type=proxy`. In `type: direct` mode, no proxy pod exists to validate OIDC tokens or emit telemetry. A CEL validation rule on `MCPRemoteEndpointSpec` SHOULD reject these fields for `type: direct`:

```cel
rule: "self.type != 'direct' || (!has(self.oidcConfig) && !has(self.telemetryConfig))"
message: "oidcConfig and telemetryConfig are not applicable when type is direct"
```

**Controller changes**: Same resolution, merge, and fail-closed logic as MCPServer (07-B), only for `type: proxy`. The resolution code should be extracted into shared helper functions to avoid duplication:
```go
func resolveOIDCConfig(ctx context.Context, client client.Client, ref *OIDCConfigWithRef, workloadKind, namespace, name string) (*ResolvedOIDCConfig, error)
func resolveTelemetryConfig(ctx context.Context, client client.Client, ref *TelemetryConfigWithRef) (*ResolvedTelemetryConfig, error)
```

**Config controller watch updates**: MCPOIDCConfig and MCPTelemetryConfig controllers (02-C, 03-C) must also watch MCPRemoteEndpoint resources for reference tracking.

## Testing

**Unit tests**:
- Same test matrix as MCPServer (07-B): happy path ref resolution, inline fallback, fail-closed on missing/non-Ready refs
- Verify shared resolution helpers work correctly for MCPRemoteEndpoint

**Integration tests** in `cmd/thv-operator/test-integration/`:
- Create MCPRemoteEndpoint (`type: proxy`) referencing MCPOIDCConfig + MCPTelemetryConfig → verify Ready
- Fail-closed: MCPRemoteEndpoint (`type: proxy`) referencing non-existent config → verify Failed phase
- CEL rejection: MCPRemoteEndpoint (`type: direct`) with `oidcConfig` set → rejected by admission

## Acceptance Criteria

- [ ] MCPRemoteEndpoint spec has `oidcConfig` and `telemetryConfig` ref+inline union fields
- [ ] CEL validation rejects `oidcConfig` and `telemetryConfig` when `spec.type=direct`
- [ ] Controller resolves config refs with same logic as MCPServer (only for `type: proxy`)
- [ ] Fail-closed on missing or non-Ready config refs (proxy mode only)
- [ ] Resolution code shared with MCPServer controller (no duplication)
- [ ] MCPOIDCConfig and MCPTelemetryConfig controllers watch MCPRemoteEndpoint
- [ ] Unit and integration tests cover ref resolution and fail-closed for MCPRemoteEndpoint
- [ ] Tests pass in CI (`task test`)
