# Add CEL Validation Rule to `OIDCConfigRef`

**Parent**: [STORY-01](STORY-01.md) — CEL Validation on Existing Union Types

## Description

Add a `+kubebuilder:validation:XValidation` marker to the `OIDCConfigRef` struct in `mcpserver_types.go` that enforces exactly one of `kubernetesServiceAccount`, `configMapRef`, or `inline` is set.

## Implementation Notes

**Pattern to follow**: `MCPExternalAuthConfig` CEL rules at `mcpexternalauthconfig_types.go:44-51`:
```go
// +kubebuilder:validation:XValidation:rule="self.type == 'tokenExchange' ? has(self.tokenExchange) : !has(self.tokenExchange)"
```

**Target struct**: `OIDCConfigRef` in `cmd/thv-operator/api/v1alpha1/mcpserver_types.go` (lines 422-448).

**CEL rule**:
```go
// +kubebuilder:validation:XValidation:rule="(has(self.kubernetesServiceAccount) ? 1 : 0) + (has(self.configMapRef) ? 1 : 0) + (has(self.inline) ? 1 : 0) == 1",message="exactly one of kubernetesServiceAccount, configMapRef, or inline must be set"
```

**Key files**:
- `cmd/thv-operator/api/v1alpha1/mcpserver_types.go` — struct definition
- Run `task gen` after changes to regenerate manifests

## Testing

Integration tests in `cmd/thv-operator/test-integration/` covering all `OIDCConfigRef` permutations:

- Valid: only `inline` set → accepted
- Valid: only `configMapRef` set → accepted
- Valid: only `kubernetesServiceAccount` set → accepted
- Invalid: both `inline` and `configMapRef` set → rejected
- Invalid: both `inline` and `kubernetesServiceAccount` set → rejected
- Invalid: both `configMapRef` and `kubernetesServiceAccount` set → rejected
- Invalid: all three set → rejected
- Invalid: none set → rejected

Tests should apply manifests against the envtest API server and assert the expected validation error messages.
